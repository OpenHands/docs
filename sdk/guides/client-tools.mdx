## Overview

**Client-defined tools** let frontend applications (e.g. a browser UI) register tools purely via JSON spec — without writing a Python executor. When the agent invokes such a tool, the SDK immediately returns an acknowledgment observation so the agent loop continues, and emits an `ActionEvent` over the WebSocket so the client can perform the actual work (open a panel, navigate, display a notification, etc.).

This is the recommended pattern for any tool whose logic lives entirely in the UI rather than on the server.

## How It Works

1. **Define** a `ClientToolSpec` — a JSON Schema description of the tool name, description, and parameters.
2. **Pass** the specs to `Conversation(..., client_tools=[...])`. The SDK registers them and injects them into the agent's tool list automatically — no manual `register_tool()` / `Tool(name=...)` wiring required.
3. **Intercept calls** in an event callback by checking `ActionEvent.tool_name`.
4. **Execute** the action in the frontend — the SDK returns a no-op acknowledgment so the agent continues.

## Defining a ClientToolSpec

```python icon="python" wrap
from openhands.sdk.tool import ClientToolSpec

spec = ClientToolSpec(
    name="show_notification",
    description="Display a notification banner in the UI.",
    parameters={
        "type": "object",
        "properties": {
            "message": {"type": "string", "description": "Text to display"},
            "level": {
                "type": "string",
                "enum": ["info", "warning", "error"],
                "description": "Severity level",
            },
        },
        "required": ["message", "level"],
    },
)
```

`ClientToolSpec` accepts any valid [JSON Schema object](https://json-schema.org/understanding-json-schema/reference/object) as `parameters`.

## Wiring Into a Conversation

Pass the specs straight to `Conversation` via `client_tools`. The SDK registers
them and injects the matching `Tool` specs into the agent — you don't need to
call `register_tool()` or add `Tool(name=...)` entries yourself.

```python icon="python" wrap
from openhands.sdk import Tool

agent = Agent(
    llm=llm,
    tools=[Tool(name=TerminalTool.name)],  # standard tools only
)

conversation = Conversation(
    agent=agent,
    workspace=cwd,
    callbacks=[on_event],
    client_tools=[spec],  # registered + injected by the SDK
)
```

<Note>
Each client tool's JSON Schema travels with the conversation, so two
conversations can define the same tool name with the same schema without
clobbering each other. Reusing a name with a *different* schema in the same
process raises an explicit error.
</Note>

## Intercepting Calls in the Frontend

Use an event callback to observe when the agent calls a client tool:

```python icon="python" wrap
from openhands.sdk import Event
from openhands.sdk.event.llm_convertible.action import ActionEvent

CLIENT_TOOL_NAMES = {"show_notification", "navigate_to"}

def on_event(event: Event) -> None:
    if isinstance(event, ActionEvent) and event.tool_name in CLIENT_TOOL_NAMES:
        args = event.action.model_dump() if event.action else {}
        # In a real app: forward args over WebSocket to the browser
        print(f"[CLIENT] {event.tool_name}({args})")

conversation = Conversation(
    agent=agent, workspace=cwd, callbacks=[on_event], client_tools=[spec]
)
```

The SDK returns `"Tool call dispatched to client."` as the observation, so the agent continues its reasoning loop without waiting for client-side execution.

## Annotations and Security Risk

By default, a `ClientToolSpec` leaves [tool annotations](https://modelcontextprotocol.io/specification/server/tools#tool)
**unset**. Client tools can trigger arbitrary frontend side effects, so the SDK
does *not* optimistically assume they are read-only or idempotent. Leaving
annotations unset keeps the conservative behavior — in particular, the agent is
asked to predict a `security_risk` before calling the tool.

If a tool really is read-only, set the annotations explicitly:

```python icon="python" wrap
from openhands.sdk.tool import ToolAnnotations

spec = ClientToolSpec(
    name="view_panel",
    description="Open a read-only panel in the UI.",
    annotations=ToolAnnotations(readOnlyHint=True, destructiveHint=False),
)
```

The original JSON Schema you provide in `parameters` (including `enum`s, nested
objects, numeric bounds, and `additionalProperties`) is preserved verbatim when
the tool is exported to the LLM, so client-defined constraints are never dropped.

## With the Agent Server

When using a remote `openhands-agent-server`, pass `client_tools` directly to `Conversation`. The server automatically registers the specs and injects them into the agent's tool list — no local registration step is needed. `ActionEvent`s are delivered to your callbacks over WebSocket, exactly as a browser frontend would receive them.

```python icon="python" wrap
conversation = Conversation(
    agent=agent,           # tool list managed by server
    workspace=workspace,   # points to agent-server
    callbacks=[on_event],  # receives ActionEvents via WebSocket
    client_tools=SPECS,    # sent in POST /api/conversations
)
assert isinstance(conversation, RemoteConversation)
```
