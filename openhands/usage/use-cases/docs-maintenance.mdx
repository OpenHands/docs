Documentation goes stale. Code evolves, APIs change, features get deprecated—but docs often lag behind. Outdated documentation is worse than no documentation because it actively misleads. What if documentation could automatically update itself when code changes?

## The Challenge

Documentation debt accumulates because:

- **Code changes faster than docs**: Features ship, but doc updates get deprioritized
- **No enforcement**: Unlike tests, outdated docs don't break the build
- **Distributed ownership**: No one owns keeping docs current
- **Manual process**: Updating docs is tedious and easy to forget

**What if we could detect and fix documentation drift automatically?**

## Automated Documentation with OpenHands

OpenHands helps keep documentation current by:

- **Detecting drift**: Identifying docs that don't match the code
- **Updating content**: Rewriting sections to reflect changes
- **Adding documentation**: Generating docs for undocumented code
- **Maintaining consistency**: Ensuring style and format match

## How to Use

### Slash Command

Use `/update-docs` to sync documentation:

```
/update-docs

The API was updated in the last PR. Please update the README
and API documentation to match the new endpoints.
```

### Automated PR Check

Run on every PR to ensure docs are updated:

```
Review this PR and check if documentation needs updating:
- README.md
- API docs in /docs/api/
- Inline code comments
- Usage examples

If any docs are outdated, update them as part of this PR.
```

## Documentation Types

### README Files

Keep project READMEs current with:
- Installation instructions
- Quick start guides
- Configuration options
- Feature descriptions

### API Documentation

Update when endpoints change:
- Request/response schemas
- Authentication requirements
- Error codes
- Example requests

### Inline Documentation

Maintain code comments:
- Function docstrings
- Complex logic explanations
- Type annotations
- Usage examples

### Tutorials and Guides

Keep learning content accurate:
- Step-by-step instructions
- Code samples that actually work
- Screenshots (flag when outdated)
- Best practices

## Example: API Documentation Update

**Code Change**:
```python
# Before
@app.route("/users/<id>")
def get_user(id):
    return User.get(id)

# After
@app.route("/users/<id>")
def get_user(id, include_profile: bool = False):
    user = User.get(id)
    if include_profile:
        user.profile = Profile.get(user.profile_id)
    return user
```

**Documentation Update**:
```markdown
## Get User

`GET /users/{id}`

### Query Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| include_profile | boolean | false | Include full profile data |

### Response

```json
{
  "id": "123",
  "name": "John Doe",
  "email": "john@example.com",
  "profile": {  // Only if include_profile=true
    "bio": "...",
    "avatar_url": "..."
  }
}
```
```

## Automation Strategies

### PR-Triggered Updates

Check and update docs on every PR:

```yaml
name: Documentation Check
on: [pull_request]

jobs:
  check-docs:
    steps:
      - name: Check for doc updates
        run: |
          # Trigger OpenHands to review and update docs
```

### Scheduled Sync

Run weekly to catch any drift:

```
Every Monday at 9am:
1. Compare code and documentation
2. Identify outdated sections
3. Create PR with updates
4. Request review from docs team
```

### On-Demand Updates

Use when you know docs need work:

```
/update-docs src/api/

Update all documentation for the API module including:
- Function docstrings
- README API section
- OpenAPI spec
```

## Best Practices

### Scope Documentation

- Focus on public APIs and interfaces
- Document non-obvious behavior
- Keep examples minimal but complete
- Update version numbers and dates

### Style Consistency

- Match existing documentation style
- Use consistent terminology
- Follow the project's doc conventions
- Include relevant links

### Verification

- Test code examples actually work
- Verify links aren't broken
- Check screenshots are current
- Validate API examples against actual behavior

## Integration with Code Review

Add to your code review process:

```markdown
## PR Checklist

- [ ] Code changes tested
- [ ] Documentation updated
- [ ] Examples verified
```

OpenHands can automatically check this:

```
Review this PR and verify:
1. All new public functions have docstrings
2. README reflects any API changes
3. Code examples in docs still work
4. No broken links in documentation
```
